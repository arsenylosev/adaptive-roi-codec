> **Назначение документа:** рабочая спецификация для проведения экспериментов и подготовки научной статьи.  
> **Авторы исследования:** Лосев А. П., Бучатский А. Н.  
> **Контекст:** работы, представленные на НТО РЭС и АПИНО 2026 (архитектура кодека, клинически ориентированная функция потерь, адаптивная квантизация по ROI).

## Аннотация (черновик для статьи)

Предлагается нейросетевой кодек видеопотока капсульной эндоскопии, объединяющий вариационный автоэнкодер (VAE) с компенсацией движения, детектор клинически значимых областей интереса (ROI) и пространственно-адаптивный квантователь латентного представления. Обучение выполняется на открытом наборе данных Kvasir-Capsule в облачной инфраструктуре Yandex Cloud; целевой сценарий развёртывания — CPU-инференс в реальном времени с ограничением по энергопотреблению. Экспериментальная программа включает абляции по параметру нелинейности κ, весам ROI-потерь и целевому битрейту; метрики качества сопоставляются с результатами предварительных публикаций авторов.

## 1. Постановка задачи

### 1.1. Научная проблема

Видеокапсула генерирует длительные эндоскопические записи (десятки часов), что создаёт проблему хранения и передачи данных. Классические кодеки не учитывают неравномерную клиническую значимость различных участков кадра: воспаления, язвы, сосудистый рисунок и локальные изменения цвета требуют более высокого качества реконструкции, чем однородные участки слизистой.

**Цель эксперимента:** реализовать, обучить и оценить нейросетевой видеокодек с **адаптивным распределением битрейта** в зависимости от обнаруженных ROI, сохраняя приемлемое качество реконструкции и удовлетворяя ограничениям по задержке и энергопотреблению на CPU.

### 1.2. Задачи исследования

| № | Задача | Связь с публикацией | Статус в репозитории |
|---|--------|---------------------|----------------------|
| 1 | VAE-кодек с компенсацией движения между кадрами | [1] | Реализован (`vae_codec.py`) |
| 2 | Клинически ориентированная функция потерь L_total | [2] | Реализована (`clinical_loss.py`) |
| 3 | Детектор ROI + адаптивная квантизация (формулы 4–6) | [3] | Реализованы (`roi_detector.py`, `quantizer.py`) |
| 4 | Обучение на Kvasir-Capsule в Yandex Cloud | §6–8 | Двухэтапный пайплайн (ADR-003) |
| 5 | CPU-инференс и измерение энергопотребления | §2.3, §7 | Запланировано (`inference.py` — вне текущего MVP) |

### 1.3. Гипотезы для проверки в статье

1. **H1 (κ):** нелинейность κ = 2,0 в зависимости q_t = f(E_ROI) обеспечивает оптимальный компромисс «качество в ROI / средний битрейт» по сравнению с κ ∈ {1,0; 1,5; 2,5}.
2. **H2 (λ_ROI):** увеличение веса λ_ROI усиливает сохранение деталей в ROI при фиксированном битрейте, но может ухудшить фоновые области.
3. **H3 (разрешение):** обучение на **нативном** разрешении Kvasir-Capsule 336×336 даёт воспроизводимые результаты без искусственного апскейла до Full HD; показатели Full HD трактуются как **целевые для этапа развёртывания** (ADR-002).
4. **H4 (инфраструктура):** двухэтапный пайплайн «извлечение кадров на CPU → обучение на предобработанных тензорах с bulk-копированием на SSD» устраняет узкое место S3 FUSE и делает загрузку GPU измеримой.

## 2. Теоретическая база и архитектура

### 2.1. Исходные публикации

| Ref | Название | Ключевой вклад |
|-----|----------|----------------|
| [1] | Development of Neural Network-based Encoder for Endoscopic Video Stream Compression | VAE: латент 120×68×192 при Full HD, 16× сжатие, skip-соединения, компенсация движения |
| [2] | Development of Clinically-Oriented Loss Function for Neural Compression of Endoscopic Video Stream | L_base, L_ROI, L_rate, L_temp; адаптивный β(t) |
| [3] | Adaptive Bitrate Allocation in Neural Network Encoder of Endoscopic Video Stream Based on ROI Detection | ROI-детектор (U-Net + MobileNetV3), κ = 2,0, пространственная маска квантования |
| [4] | Smedsrud P. H. et al. Kvasir-Capsule | Открытый датасет капсульной эндоскопии |

### 2.2. Модульная схема (для рисунка в статье)

```
Входной кадр x_t (336×336×3)
        │
        ▼
┌───────────────────┐
│  Детектор ROI     │  MobileNetV3-large + U-Net-подобная голова
│  M_t ∈ [0,1]      │  Вход/выход: 336×336
└─────────┬─────────┘
          │ M_t
          ▼
┌───────────────────┐
│  VAE-кодек        │  Encoder → z_t; Decoder + skip; prev_recon, prev_z
│  z_t: 192×21×21   │  16× пространственное сжатие (336÷16 = 21)
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│ Адаптивный        │  q_t = q_min + (q_max−q_min)·E_ROI^κ
│ квантователь      │  Q_t = q_t·(1 + α·M_t)
└─────────┬─────────┘
          ▼
   ẑ_t → (entropy coding) → битовый поток
```

**Важное уточнение для статьи:** в публикациях [1–3] заложено разрешение **1920×1080** и латент **120×68×192**. В данном эксперименте принято решение ADR-002: обучение в **нативном** разрешении Kvasir-Capsule **336×336**, латент **21×21×192**. Это должно быть явно отражено в разделе «Материалы и методы» с обоснованием (отсутствие дополнительной информации при апскейле, экономия вычислений ~32× по площади кадра).

### 2.3. Адаптивная квантизация (формулы для статьи)

| Обозначение | Формула | Значение по умолчанию |
|-------------|---------|----------------------|
| E_ROI(t) | (1/HW) Σ M_t(i,j) | доля ROI в кадре |
| q_t | q_min + (q_max − q_min) · E_ROI^κ | κ = **2,0** (основной эксперимент) |
| Q_t(p,q) | q_t · (1 + α · M_t) | α = 0,5 |
| q_min, q_max | — | 0,1; 2,0 |

Реализация: `adaptive_roi_codec/model/quantizer.py`.

### 2.4. Функция потерь (формула 14, [2])

```
L_total = L_base + λ_ROI·L_ROI + λ_rate·L_rate + λ_temp·L_temp
```

| Компонента | Определение | Параметр |
|------------|-------------|----------|
| L_base | α·(−PSNR) + (1−α)·(1−SSIM_soft) | α = 0,5 |
| L_ROI | взвешенная MSE в ROI по маске M_t | λ_ROI = 1,5 |
| L_rate | β · KL(q_φ(z\|x) ‖ N(0,I)) | λ_rate = 0,01; β_0 = 0,01 |
| L_temp | согласованность (x̂_t − x̂_{t−1}) и межкадрового движения | λ_temp = 0,1 |

Межкадровое состояние `video_states` хранит (recon, z) по `video_id` внутри эпохи для корректной компенсации движения при перемешивании батчей.

## 4. Протокол обучения

### 4.1. Двухэтапный облачный пайплайн (ADR-003)

| Этап | Job | Инстанс | Назначение |
|------|-----|---------|------------|
| **1 — Extract** | `launch-train --job extract` | CPU `c1.8` | Декодирование MP4 один раз; запись `.npy` на S3 |
| **2 — Train** | `launch-train --execute` | GPU `g1.1` (V100) / `gt4.1` (T4 smoke) | Загрузка предобработанных кадров; GPU-обучение |

**Этап 2 — устранение латентности S3:** при `stage_frames_local: true` и `stage_mode: bulk` весь набор кадров, указанный в манифесте, **копируется на extended SSD** job-а (`/servant/jobs/extended_working_storage/frame_cache/`, до 150 GB) **до** начала итераций обучения. Это переводит `data_wait` с ~2 с/батч (S3 FUSE) на чтение с локального SSD (mmap `.npy`).

### 4.2. Гиперпараметры обучения (базовая конфигурация)

Источник: `configs/base.yaml`, переопределения — `jobs/inputs/*.json`.

| Группа | Параметр | Значение |
|--------|----------|----------|
| Модель | `input_res` | [336, 336] |
| | `latent_h`, `latent_w`, `latent_ch` | 21, 21, 192 |
| Обучение | `batch_size` | 4 (base); 12 (smoke/monitor jobs) |
| | `lr` | 1×10⁻⁴ |
| | `epochs` | 50 (полный прогон); 1 (smoke) |
| | `warmup_epochs` | 5 |
| DataLoader | `num_workers` | 2 (smoke); 4 (V100 monitor) |
| | `prefetch_factor` | 2 |
| | `stage_mode` | `bulk` |
| ROI | `backbone` | mobilenet_v3_large |
| | `pretrained` | true (base); false (smoke ablation) |
| Квантователь | `kappa` | 2,0 (baseline) |

### 4.3. Программные зависимости на GPU-worker

- Python 3.11, PyTorch **2.4.1+cu121** (совместимость с драйвером DataSphere V100)
- `TRAIN_REQUIRE_CUDA=1` — fail-fast при отсутствии GPU
- Опционально: Docker-образ проекта DataSphere (`docker/datasphere-gpu/`) для исключения pip-установки на root overlay

### 4.4. Контрольные прогоны (рекомендуемая последовательность)

1. **Локальный dry-run:** синтетические кадры, проверка градиентов и loss.
2. **Smoke GPU (`train_smoke.json`):** 2400 кадров, 200 батчей, T4/V100, проверка `Device: cuda`, метрик job output.
3. **Monitor V100 (`train_smoke_v100_monitor.json`):** длинный прогон (4000 батчей, 48000 кадров) для профиля `data_wait` / `compute`, утилизации GPU/RAM.
4. **Полное обучение 50 эпох:** `train_input.json`, checkpoint каждые 5 эпох на S3.

## 6. Метрики и критерии успеха

### 6.1. Метрики качества (целевые ориентиры из [3], Full HD)

| Метрика | Целевое значение [3] | Примечание для 336×336 |
|---------|---------------------|------------------------|
| Dice ROI | ≥ 0,89 | пересчитать на val/test |
| PSNR | ≥ 32,7 dB | **перебenchmark** на native res |
| Средний битрейт | ≤ 1,81 Mbps | после entropy coding |
| Субъективная оценка | ≥ 4,6 / 5,0 | экспертная оценка врачами |
| Задержка CPU | ≤ 33 ms/кадр | этап развёртывания |
| Энергопотребление CPU | ≤ 28 W | этап развёртывания |

**Для статьи:** явно разделить таблицу «результаты обучения 336×336» и «прогноз/экстраполяция Full HD» либо провести отдельный эксперимент upscaled inference.

### 6.2. Метрики реализации (репозиторий)

Реализовано в `adaptive_roi_codec/utils/metrics.py`:

- PSNR
- Dice ROI (для сравнения масок)

Планируется для статьи: MS-SSIM, фактический битрейт после энтропийного кодирования, ROC/PR для ROI при наличии разметки.

### 6.3. Критерии приёмки облачного пайплайна

- [ ] stdout: `Device: cuda`, имя GPU
- [ ] `gpu_stats`: устойчивая утилизация GPU при bulk staging + workers > 0
- [ ] `data_wait` ≪ 2 с/батч после bulk copy
- [ ] Job status **Success**, metrics output найден DataSphere
- [ ] Checkpoint на S3 после N эпох
